# Orbital Organization Catalog Query API

Source: Cisco DevNet, "Create an Organization Catalog Query", checked 2026-06-16.

## Create

- Method: `POST`
- Path: `/v0/catalogs`
- Purpose: create an organization catalog query.

Project convention: catalog queries generated by Codex must start their title with `codex`, for example `codex - process inventory`.

Request body fields observed in DevNet schema:

```json
{
  "title": "Catalog query title",
  "description": "Catalog query description",
  "platform": ["windows"],
  "version": "1.0",
  "osquery": [
    {
      "label": "query_label",
      "name": "Readable query name",
      "sql": "SELECT * FROM processes;",
      "types": []
    }
  ]
}
```

Response status `200` returns a catalog query response object with `data` and `errors`.

Known error examples from DevNet:

- `400`: query content too large, including `query content should not exceed 65536 bytes`
- `401`: authentication required
- `403`: access to this feature is not permitted

## Disable / Remove

- Method: `DELETE`
- Path: `/v0/catalogs/{catalogid}`
- Purpose: disable/remove an organization catalog query.
- Path parameter: `catalogid`, for example an ID starting with `org:`.
- Use the raw catalog ID in the path, including the colon, for example `/v0/catalogs/org:example`. URL-encoding the colon as `%3A` caused `cannot resolve catalog query id=org%3A...` during testing.
- When the target ID is already known, the preferred flow is a direct `DELETE /catalogs/{catalogid}`. A full `GET /catalogs` before deletion is only needed when the operator wants to include title/context before deleting.

Response status `200` returns a catalog query response object with `data` and `errors`.

Known error examples from DevNet:

- `400`: catalog query not found
- `401`: authentication required
- `403`: access to this feature is not permitted

## Update / Rename

- Method: `PUT`
- Path: `/v0/catalogs`
- Purpose: update an organization catalog query. A rename is a preserved-body update where only `title` changes.
- Body: include the catalog `id` plus the catalog query fields to preserve, especially `description`, `platform`, `version`, and `osquery`.
- Fast known-ID flow: fetch the existing catalog item with `GET /catalogs/{catalogid}`, preserve `description` and `config.osquery`/`osquery`, set the same `id`, change only `title`, send `PUT /catalogs`, then verify with `GET /catalogs/{catalogid}`.
- A full `GET /catalogs` scan is only needed for duplicate-title checks or broad inventory context.

For Codex-generated entries, titles should normally start with `codex`. If the user explicitly requests a different title, preserve the requested title and record that this was an explicit user-provided title.

## Field Distinction

Catalog create/update body uses lower-case `osquery`.

Endpoint query execution body uses camelCase `osQuery` and includes execution fields such as `nodes`, `interval`, and `expiry`.

When organization catalog queries are fetched back with `GET /catalogs`, SQL blocks appear under `config.osquery`:

```json
{
  "title": "Catalog query title",
  "config": {
    "description": "Catalog query description",
    "osquery": [
      {
        "label": "query_label",
        "sql": "SELECT * FROM processes;"
      }
    ]
  }
}
```

Do not use this as the default create body. In testing, `POST /catalogs` with explicit `config.osquery` returned `No sql statements found`. Use the DevNet top-level `osquery` create body, then expect readback under `config.osquery`.

For `POST /catalogs`, keep each SQL block as a single-line string. Multiline SQL can produce server-side GraphQL errors such as `literal not terminated`, while the same SQL succeeds after whitespace normalization.

## Safety

Creating a catalog query does not run it on endpoints. Endpoint execution happens later through the UI or query API when targets are supplied.
