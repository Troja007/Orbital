# Orbital Update Catalog

Use this skill for write-capable Orbital Catalog query work. Keep it separate from endpoint query execution: this skill creates, previews, or removes organization catalog query definitions, while `orbital-run-osquery-live-query` runs queries against endpoints.

## Core Rule

Do not execute endpoint queries from this skill. Catalog creation saves reusable query content only.

Any catalog query generated by Codex must use a title that starts with `codex`. Prefer lower-case `codex - <purpose>` for titles and `codex_<purpose>` for SQL block labels.

## API Shape

The DevNet operation **Create an Organization Catalog Query** uses:

- Method/path: `POST /v0/catalogs`
- Required request body concept: catalog query record
- Important request fields:
  - `title`: catalog query title
  - `description`: catalog query description
  - `platform`: array such as `["windows"]`
  - `version`: string
  - `osquery`: array of labeled SQL blocks

Each `osquery` item should contain:

- `label`: stable label used to identify the result block
- `name`: optional readable name
- `sql`: osquery SQL string
- `types`: optional CTIM observable type annotations

Orbital stores organization catalog SQL blocks under `config.osquery` when catalog entries are read back from `GET /catalogs`. For `POST /catalogs`, use the DevNet top-level `osquery` body. Keep SQL strings single-line for creation; multiline SQL can trigger server-side GraphQL literal parsing errors even when the JSON body is valid.

Do not confuse either catalog shape with endpoint execution bodies, which use `nodes`, `interval`, `expiry`, and camelCase `osQuery`.

The DevNet operation **Disable an Organization Catalog Query** uses:

- Method/path: `DELETE /v0/catalogs/{catalogid}`
- ID shape: organization catalog query IDs normally start with `org:`
- Use the raw `org:` ID in the path. Do not URL-encode the colon.
- If the catalog ID is already known, delete directly by ID. Do not fetch the full catalog before deletion unless title/context is needed.

The DevNet operation **Update an Organization Catalog Query** uses:

- Method/path: `PUT /v0/catalogs`
- Required request body concept: catalog query record with `id`
- Important request fields: `id`, `title`, `description`, `platform`, `version`, and `osquery`

For a known-ID rename, use the fast path: fetch the existing entry with `GET /v0/catalogs/{catalogid}`, preserve its SQL/config, change only `title`, include the same catalog `id` in the body, send `PUT /v0/catalogs`, then verify with `GET /v0/catalogs/{catalogid}`. Avoid a full `GET /v0/catalogs` scan unless duplicate-title checking is explicitly needed.

## Workflow

1. Build or inspect a catalog request JSON file.
2. Validate:
   - title is present
   - Codex-generated titles start with `codex`
   - description is meaningful
   - platform is present
   - each SQL block has `label` and `sql`
   - SQL statements end with semicolons where practical
   - body size is below the documented catalog content limit
3. Preview the JSON body first.
4. Check existing organization catalog queries for duplicate titles.
5. Upload only when the user explicitly asked to create/update the catalog entry.
6. Store local operational metadata under `local/orbital_catalog_updates/`; do not store bearer tokens.
7. Do not store generated catalog payloads or SQL in GitHub-tracked folders unless the user explicitly asks to save the query as custom repository content.

## Helper

Use the bundled helper:

```bash
python3 skills/orbital-update-catalog/scripts/update_catalog_query.py \
  --input-json local/generated-queries/example_catalog_query.json \
  --region us \
  --preview
```

Create the catalog entry:

```bash
python3 skills/orbital-update-catalog/scripts/update_catalog_query.py \
  --input-json local/generated-queries/example_catalog_query.json \
  --region us \
  --create
```

The helper rejects non-`codex` titles by default. Use `--allow-non-codex-title` only when the user explicitly provides a non-Codex catalog title and asks to preserve it.

Rename a catalog entry while preserving the existing SQL blocks:

```bash
python3 skills/orbital-update-catalog/scripts/update_catalog_query.py \
  --region us \
  --rename-catalog-id org:<catalog_id> \
  --new-title "new title"
```

For Codex-generated entries, titles should normally start with `codex`. If the user explicitly requests a non-`codex` title, pass `--allow-non-codex-title` and preserve the requested title.

Known-ID rename is optimized by default and skips duplicate-title scans. Use `--check-duplicate-title` only when duplicate detection is more important than speed.

Remove/disable catalog entries:

```bash
python3 skills/orbital-update-catalog/scripts/update_catalog_query.py \
  --region us \
  --delete-catalog-id org:<catalog_id>
```

The default delete flow is direct and fast: `DELETE /catalogs/{catalogid}` followed by a post-delete verification lookup. Use `--describe-before-delete` only when you need the helper to fetch the full catalog first and include the entry title/context in the output. Use `--no-verify-delete` only when the DELETE response is enough and the fastest possible path is required.

Multiple IDs can be removed in one helper call by repeating `--delete-catalog-id`.

If a source SQL is multiline, normalize whitespace before upload:

```bash
python3 - <<'PY'
import json, re
from pathlib import Path
p = Path("catalog_query.json")
body = json.loads(p.read_text())
for item in body["osquery"]:
    item["sql"] = re.sub(r"\s+", " ", item["sql"]).strip()
p.write_text(json.dumps(body, indent=2))
PY
```

The helper reads credentials from environment variables or `02_Working_Files/orbital_credentials.env`:

- `ORBITAL_API_TOKEN`, `ORBITAL_TOKEN`, `SECUREX_TOKEN`, `CISCO_SECUREX_TOKEN`, or `CISCO_TOKEN`
- or `ORBITAL_CLIENT_ID` plus `ORBITAL_CLIENT_SECRET`
- `ORBITAL_REGION`, defaulting to `eu` if not supplied

## Duplicate Handling

By default, the helper checks `GET /v0/catalogs` and stops before creation if a catalog query with the same title already exists. Use a new title or intentionally update through a dedicated update workflow after verifying the target catalog ID.

Do not overwrite or disable catalog entries unless the user explicitly asks for that operation and the catalog ID has been verified.
