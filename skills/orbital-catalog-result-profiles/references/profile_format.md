# Catalog Result Profile Format

Use this reference when changing the generated result-profile structure or reviewing generated Markdown files.

## Purpose

Each per-Catalog-ID Markdown file should help an incident responder answer:

- What does this Orbital catalog query answer?
- What does a returned row mean?
- What does zero rows mean?
- What does no endpoint response mean?
- What assumptions are safe?
- What assumptions are unsafe?
- What follow-up is appropriate?

Do not optimize these files for developers only. They are analyst-facing explanation artifacts.

## Required Per-Query Sections

Generate each file with these sections:

1. H1 catalog query title
2. `Catalog Identity`
3. `What This Query Answers`
4. `Expected Result Shape`
5. `How To Read The Result`
6. `Incident Responder Assumptions`
7. `Caveats`
8. `Recommended Follow-Up`
9. `Explanation Template`
10. `Privacy Boundary`

## Result Reading Rules

Separate these states clearly:

- Endpoint answered with rows: interpret returned rows according to catalog purpose, categories, columns, and warnings. Rows are evidence, not automatically a malicious verdict.
- Endpoint answered with zero rows: read as no matching rows for the exact query scope. This is not a query failure and not proof the behavior never happened.
- Endpoint did not respond: do not treat as zero rows or clean. Check target selection, endpoint online state, job status, result retrieval, and API errors before rerunning.
- Completed with label or endpoint errors: read error fields before interpreting row counts.

## Filename Rule

Use:

```text
<catalog_id>__<normalized_query_name>.md
```

Normalization:

- lowercase
- ASCII letters and digits
- replace non-alphanumeric runs with `_`
- trim leading/trailing `_`

Example:

```text
process_hashes__sha256_hash_of_running_processes.md
```

## Machine And Human Files

Keep both forms:

- `windows_stock_catalog_result_profiles.jsonl`: machine-readable index and automation source.
- `windows_stock_catalog_result_profiles.md`: human-readable summary table.
- `windows/*.md`: analyst-facing per-query explanation files.

The JSONL and per-query Markdown should be generated by the same helper so they stay consistent.

## Privacy Boundary

Allowed:

- Catalog metadata
- Sanitized validation status
- Row counts and buckets from the validation run
- Returned label names
- Returned column names
- Sanitized error classes
- Generic target scope, such as "one explicit Windows endpoint"

Forbidden:

- Endpoint result rows
- Hostnames
- Target selectors such as `host:<value>`
- Orbital Job IDs / `orbital_queryID`
- Usernames
- IP addresses
- GUIDs
- Tenant or customer identifiers
- Raw API responses
- Credentials, bearer tokens, client IDs, or secrets


If validation finds sensitive values, stop before staging. Fix the generator or generated files, rerun generation, and validate again.
